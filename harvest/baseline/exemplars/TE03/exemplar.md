Suppose you have a million user records and need to find one by email address. Stored in an array, the records force a search: start at the front, compare each entry, stop at the match. On average you inspect half a million records. Sorting the array lets binary search cut that to about twenty comparisons, but now every insertion has to shift entries around to preserve the order. The basic structures make you choose between fast lookup and fast insertion.

A hash table refuses the choice. The idea is to compute where a record lives instead of searching for it. You allocate an array of, say, 1,024 slots. Then you write a hash function, a routine that takes a key such as the email address and grinds it down to a number. A crude version might sum the character codes; real ones mix the bits more thoroughly. Take that number modulo 1,024 and you have an array index. To store a record, hash its key and put the record in the slot the hash names. To find it later, hash the key again. The same input produces the same number, the same number names the same slot, and the record is sitting there. No search happened. Whether the table holds a thousand records or a hundred million, the cost is one computation and one array access.

There is a catch, and it has a name: collision. Two different keys can hash to the same slot, and once you have more keys than slots, some pairs must. The standard fix is to let each slot hold a short list instead of a single record. Lookup becomes: hash the key, go to the slot, walk the list. If the hash function spreads keys evenly, each list stays short, one or two entries, and walking it costs almost nothing.

That "if" is carrying weight. A poor hash function piles keys into a few slots, the lists grow long, and you are back to searching. The same thing happens when the table simply fills up. Implementations therefore track the load, the ratio of records to slots, and when it passes a threshold they allocate a bigger array and rehash everything into it. The occasional rebuild is the price of keeping every ordinary operation cheap.

That is the whole trade: you give up ordering, since records land wherever the hash scatters them, and in exchange lookup, insertion, and deletion all run in constant time on average. Every dictionary, map, and object property lookup you have ever used is this trick underneath.
